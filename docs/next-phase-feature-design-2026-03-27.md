# 下一阶段功能设计案（Cloudflare 托管场景）

> 日期：2026-03-27  
> 目标：先做“用户可感知价值高”的功能页与流程，逐步从基础系统走向大型期刊平台。

---

## 0. 设计原则

1. **先做“可见价值”**：优先上线能被作者/审稿人/编辑直接感知的页面和能力。
2. **不打断现有主流程**：投稿、编辑、审稿链路保持稳定，新增功能以“并行增强”方式接入。
3. **Cloudflare 友好**：
   - 静态内容优先边缘缓存；
   - 需要鉴权的数据走 API 动态渲染；
   - 大文件与长任务尽量异步（队列/后台处理）。
4. **每个新页面都必须可运营**：有埋点、有状态、有错误提示、有回退路径。

---

## 1. 产品目标（未来 2~3 个迭代）

### 目标 A：让“作者体验”像成熟期刊
- 看得见稿件状态与时间线；
- 能方便提交修订与补充材料；
- 能一键导出引用与元数据。

### 目标 B：让“编辑效率”明显提升
- 统一工作台（待办、逾期、风险）；
- 批量操作能力；
- 审稿邀请与催办模板化。

### 目标 C：让“品牌可信度”可对外展示
- 公开统计页面；
- 透明政策与版本历史；
- 伦理案件与更正机制可被追踪。

---

## 2. 新页面与功能（优先级顺序）

## P1（建议先做，4~6 周）

### 页面 1：作者稿件时间线页（`/account/submissions/[id]/timeline`）
**核心价值**：作者可直观看到稿件从投稿到出版的关键节点。  
**模块设计**：
- 时间线节点：submitted → under_review → revision_requested → accepted/rejected → production → published
- 每个节点包含：时间、操作者角色、备注摘要。
- 节点异常提醒：如“超过 SLA 未推进”。

**数据来源**：
- `audit_logs`、`submission_versions`、`review_assignments`、`editor_decisions`、`production_jobs`。

---

### 页面 2：修订中心（`/account/submissions/[id]/revision-center`）
**核心价值**：标准化“返修”流程，减少邮件往返。  
**模块设计**：
- 上传修订稿 + 回复信（response to reviewers）；
- 自动生成版本号（v2/v3）；
- 差异摘要（作者填写）。

**需要新增能力**：
- 返修截止时间与自动提醒；
- 返修提交后自动通知负责编辑。

---

### 页面 3：编辑待办工作台（`/editor/inbox`）
**核心价值**：把“分散的列表”变成“按优先级处理队列”。  
**模块设计**：
- 分组：待分配审稿人 / 等待审稿意见 / 待决策 / 生产中异常。
- 快捷动作：分配审稿、发送催办、做决定。
- 风险标签：超时、高风险 IP、材料不合规。

---

### 页面 4：审稿邀请管理页（`/editor/reviewer-pool`）
**核心价值**：提高邀请命中率与按时返回率。  
**模块设计**：
- 审稿人画像（研究方向、历史响应时长、按时率）；
- 邀请模板 + 一键发送；
- 自动二次邀请策略（未响应 N 天自动补邀）。

---

## P2（中期增强，6~10 周）

### 页面 5：公开统计与透明度页（`/community/metrics`）
**核心价值**：建立外部信任。  
**展示指标建议**：
- 平均首轮决定时长；
- 接收率、拒稿率、撤稿/更正数量；
- 学科分布与文章类型分布。

---

### 页面 6：政策中心（`/policies`）
**核心价值**：把审稿政策、伦理政策、数据政策版本化并公开。  
**模块设计**：
- 每份政策有版本号/生效日期/变更说明；
- 可引用永久链接（便于投稿须知中引用）。

---

### 页面 7：伦理案件公开摘要页（`/ethics/transparency`）
**核心价值**：形成规范化纠错文化。  
**模块设计**：
- 不展示隐私细节，只展示案件类型、状态、处理结果与时间；
- 支持按年份/类型筛选。

---

## 3. API 与数据模型增量设计

### 3.1 API（新增建议）
- `GET /api/submissions/:id/timeline`：聚合时间线数据。
- `POST /api/submissions/:id/revision-submit`：作者提交返修。
- `GET /api/editor/inbox`：编辑待办聚合。
- `GET /api/editor/reviewer-pool`：审稿人画像与建议。
- `GET /api/public/metrics`：公开统计指标。
- `GET /api/public/policies`：政策列表与版本。

### 3.2 数据表（新增建议）
- `reviewer_profiles`：审稿人能力与历史统计。
- `revision_deadlines`：返修截止、提醒状态。
- `policy_documents`：政策正文、版本、发布日期。
- `public_metrics_snapshots`：每日统计快照（用于稳定展示）。

---

## 4. Cloudflare 场景下的页面渲染建议

- **强缓存页面**（可 ISR/静态）：
  - `/community/metrics`、`/policies`、`/ethics/transparency`
- **鉴权动态页面**（SSR/API 实时）：
  - `/editor/inbox`、`/editor/reviewer-pool`、`/account/submissions/[id]/timeline`
- **上传与长任务**：
  - 前端只负责提交；
  - 后端异步处理（扫描、索引导出、通知重试）。

---

## 5. 实施顺序（建议）

### Sprint 1（2 周）
1. 作者稿件时间线页 + `GET /api/submissions/:id/timeline`。
2. 编辑待办工作台 MVP + `GET /api/editor/inbox`。
3. 基础埋点与操作审计补全。

### Sprint 2（2 周）
1. 修订中心 MVP + 返修提交 API。
2. 审稿邀请管理页 MVP（模板 + 一键邀请）。
3. 自动提醒机制（返修截止/审稿逾期）。

### Sprint 3（2 周）
1. 公开统计页 + 每日快照任务。
2. 政策中心（版本化）。
3. 伦理透明页（脱敏摘要）。

---

## 6. 验收标准（每个页面都要满足）

- 有空态、错误态、加载态；
- 有审计日志；
- 有权限检查；
- 有埋点事件（点击、提交、失败）；
- 有回滚方案（关闭开关后不影响主流程）。
