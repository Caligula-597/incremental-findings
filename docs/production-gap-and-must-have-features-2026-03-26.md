# 已上线版本的缺口清单（必须补齐 vs 高回报提升）

> 时间：2026-03-26  
> 目标：回答“现在还差什么、哪些功能必须加、哪些提升最大”。

## 一句话结论
你们现在已经不是“功能没做完”的问题，而是进入了**生产化深水区**：
- **必须补齐（P0）**：不补会直接影响安全、稳定性、可追责。
- **高回报增强（P1）**：补完后会显著提升编辑效率、审稿质量和对外可信度。

---

## P0：必须补齐（建议 2~4 周内完成）

### 1. 分布式限流 + 统一风控阈值中心
**为什么必须加**：当前单机内存限流在多实例下会失效或被绕过。  
**最小可交付**：
- 限流迁移到 Redis/Upstash（按 IP + 账户 + 路由多维桶）。
- 把认证、编辑后台、文件下载、导出接口纳入统一策略。
- 增加“误封恢复”操作与审计日志。

### 2. 会话可吊销（Session Revocation）
**为什么必须加**：仅靠 cookie 过期无法应对凭据泄露后的紧急下线。  
**最小可交付**：
- token 增加 `sid`。
- 服务端维护会话状态（active/revoked/expired）。
- 支持“单设备下线”和“全账号强制下线”。

### 3. 文件安全：上传扫描 + 类型白名单 + 对象存储隔离
**为什么必须加**：投稿平台天然是文件入口，需防木马、脚本载荷和伪装文件。  
**最小可交付**：
- 上传后异步病毒扫描（ClamAV/SaaS 扫描服务）。
- MIME + 扩展名双重校验。
- 存储桶私有化、签名 URL 最短时效、下载审计。

### 4. 可靠通知链路（邮箱/消息）
**为什么必须加**：审稿与编辑协作高度依赖通知，丢信会直接拖慢流程。  
**最小可交付**：
- 通知表加入 `status/retry_count/next_retry_at/provider_message_id`。
- 指数退避重试 + 死信队列。
- 模板版本化与发送结果回写。

### 5. 灾备与恢复演练（Backup/Restore Drill）
**为什么必须加**：真正上线后，恢复能力比“有备份”更重要。  
**最小可交付**：
- 数据库每日备份 + 每周恢复演练。
- RTO/RPO 目标明确（例如 RTO 4h / RPO 15min）。
- 演练结果进入运维周报。

---

## P1：高回报提升（建议 1~2 个迭代）

### 6. 可观测性三件套：日志、指标、告警
- 结构化日志（请求 ID、submission_id、user_id、route、latency）。
- 指标看板：注册成功率、投稿成功率、审稿 SLA、5xx 比例。
- 告警规则：认证失败突增、队列堆积、导出失败率异常。

### 7. 审稿流程增强（对业务价值很高）
- Reviewer 候选池评分（研究方向、历史准时率、冲突关系）。
- 自动提醒（到期前 7/3/1 天）和逾期升级策略。
- 二轮审稿（major revision）模板化路径。

### 8. 编辑台效率工具
- 批量操作（批量分配、批量催办、批量导出）。
- Saved Views（按学科/状态/风险分数保存筛选）。
- Timeline 可视化（从投稿到上线所有关键节点）。

### 9. 对外可信度能力
- 公开统计页（处理时长、接收率、撤稿率、伦理案件摘要）。
- 版本化政策中心（审稿政策/伦理政策发布时间线）。
- DOI 与元数据对接完善（Crossref/DataCite 正式流程）。

---

## P2：中长期壁垒（有资源再上）

### 10. 反学术不端能力
- 文本重复度检测接口。
- 图像篡改检测接口。
- 参考文献一致性与格式自动检查。

### 11. 生态互联
- ORCID 深度联动（自动回填作者资料）。
- 机构 SSO/SAML（高校与研究机构合作常见诉求）。
- 开放 API key + webhook 事件订阅。

---

## 你现在最该做的“前三件事”（按投入产出比排序）

1. **把限流和会话治理做成“可在事故时快速止血”的体系**（P0-1, P0-2）。
2. **把文件上传安全和通知可靠性补齐**（P0-3, P0-4）。
3. **建好基础监控与告警，让团队先“看得见问题”**（P1-6）。

---

## 决策建议（给你这个阶段）

如果你们准备在未来 1~2 个月内承接真实投稿量：
- **P0 项建议全部列为上线门槛**（没有就不扩量）。
- P1 至少完成 6 + 7（可观测性 + 审稿流程增强）中的核心子集。

这样做完后，平台会从“功能可用”升级到“**可稳定运营**”。
