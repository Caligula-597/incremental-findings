## 风险与改进建议（按优先级）

## P0（建议立即处理）

### 1) 限流存储为进程内 Map，横向扩展后保护会显著弱化
- 现状：限流状态存储在内存 `Map`，实例重启即丢失，多实例之间不共享。
- 风险：在生产多副本/Serverless 场景，攻击者可“分摊打点”绕过单实例限流。
- 建议：
  - 将限流 bucket 迁移到 Redis/Upstash 或 Postgres 原子计数。
  - 统一引入 `x-forwarded-for` 信任链策略（仅在网关层注入并校验）。

### 2) Session Token 仅签名无服务端会话态，不便吊销与强制失效
- 现状：cookie 中存放签名后的用户 payload；依赖 cookie 过期控制生命周期。
- 风险：若 token 泄露，在有效期内难以服务端即时吊销；缺少 `iat`/`exp`/`sid` 精细治理。
- 建议：
  - 增加 `sid`（会话 ID）并在服务端维护 allowlist/denylist。
  - 增加短时 access token + 可轮换 refresh 机制，支持安全事件触发全局下线。

## P1（建议本迭代处理）

### 3) 生产环境仍可能退回 memory 模式，需“默认失败”而非“默认降级”
- 现状：`SUPABASE_URL`/`SUPABASE_SERVICE_ROLE_KEY` 缺失时，会返回 `null` client 并由上层走内存分支。
- 风险：误配置时可能出现“看起来可用，实则数据未持久化/多实例不一致”。
- 建议：
  - 生产默认启用 `REQUIRE_SUPABASE=true`。
  - 在关键写路径（auth/submission/editor）统一 `assertSupabaseAvailability`，避免静默降级。

### 4) 登录查询路径可优化（双查询）
- 现状：登录时会分别按 email 与 username 查询。
- 风险：在高并发下带来额外数据库开销；错误处理分支复杂度提高。
- 建议：
  - 增加统一规范化标识字段或用 `or` 查询；并加命中索引。

## P2（可排期优化）

### 5) 安全事件与风控策略可进一步“产品化”
- 建议：
  - 对 `risk_check`、`blocked` 等事件追加告警渠道（Webhook/邮件/Slack）。
  - 明确误报处置流程、阈值回归策略、留存与归档周期 SLO。
